------------------------------------------------------
POR QUÉ EL AUTO-DEPLOY DE GITHUB NO FUNCIONA (Y NO ES UN BUG)
------------------------------------------------------
Vercel plan Hobby (gratis) exige que la cuenta de GitHub conectada a Vercel sea la
DUEÑA del repositorio (o tenga la app de Vercel instalada como tal). Como hay más de
una persona/identidad escribiendo código en este repo (Luciano + colaborador + agentes
de IA), Vercel lo trata como "colaboración" y no lo permite en Hobby. Ser colaborador
con rol Admin en el repo NO alcanza — GitHub Apps instaladas por una cuenta solo pueden
acceder a repos que esa cuenta posee, no a repos de otra persona aunque seas admin ahí.

Esto NO se arregla:
  - verificando el email en GitHub (ya se probó, no alcanza)
  - dándole rol de Admin a un colaborador en el repo (ya se probó, no alcanza)
  - creando una cuenta de Vercel nueva con otro email (mismo límite, se repite igual)

Esto SÍ se arregla:
  - Actualizando el equipo de Vercel a plan Pro (~USD 20/mes) — permite múltiples
    colaboradores y reconecta el auto-deploy de GitHub sin restricciones.
  - O seguir deployando manual por CLI como ahora (gratis, funciona bien, solo
    requiere un comando después de cada cambio).

------------------------------------------------------
SI UN DEPLOY QUEDA "BLOCKED" (commit email no coincide con cuenta de GitHub)
------------------------------------------------------
A veces `vercel deploy --prod --yes` queda en estado BLOCKED con el mensaje
"The deployment was blocked because the commit email ... could not be matched
to a GitHub account." Es intermitente — no depende de tener o no Git conectado
(se probó sin Git conectado y bloqueó igual).

Workaround que funciona siempre (probado en sesión 25): hacer que el autor del
commit que se va a deployar coincida con el email de la cuenta de Vercel que
deploya (nalux2430@gmail.com), SOLO localmente, sin pushear ese cambio:

    git commit --amend --author="nalux2430-3969 <nalux2430@gmail.com>" --no-edit
    npx vercel deploy --prod --yes
    git reset --hard <sha original del commit>   # vuelve todo a como estaba, sin tocar GitHub

Importante: el `git reset --hard` final es obligatorio — sin eso el repo local
queda con un commit reescrito que no coincide con lo pusheado a GitHub.
